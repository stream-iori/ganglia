# Ganglia 需求文档 (Requirements)

> **状态**：草案 / 需求定义
> **版本**：0.1.0

## 1. 功能需求 (Functional Requirements)

### 1.1 核心 Agent 循环 (ReAct)
#### 1.1.1 正常流程
*   **输入处理**：系统应接受来自用户的自然语言输入。
*   **推理 (Thought)**：Agent 在采取行动前应生成“思考”，解释其推理过程。
*   **工具选择**：Agent 应根据用户请求从可用工具集中选择合适的工具。
*   **执行**：系统应执行所选工具并捕获其输出（观察 - Observation）。
*   **迭代**：ReAct 循环应持续进行（思考 -> 工具 -> 观察），直到得出最终答案或满足终止条件。
*   **响应**：系统应向用户提供最终的自然语言响应。

#### 1.1.2 异常处理
*   **结构化工具失败**：系统应通过 `ToolErrorResult`（如超时、超出大小限制、命令失败）捕获结构化的工具执行错误，并将详细上下文反馈给 Agent 以进行自我修正。
*   **内存保护**：系统应对所有工具执行强制执行最大输出大小（如 8KB），以防止内存耗尽和上下文窗口溢出。
*   **早期验证**：直接读取文件的工具应在将内容加载到内存之前，先根据元数据验证文件大小。
*   **循环限制**：系统应强制执行最大迭代次数（如 20 步），以防止无限循环。

### 1.2 工具系统（“双手”）
*   **内置工具（标准库）**：
    *   **本地文件系统（基于 Bash）**：优化的系统命令（`list_directory`, `read_file`, `grep_search`, `glob`）和通用的 `run_shell_command`，带有强制超时和 8KB 输出大小限制。
    *   **本地文件系统（基于 Vert.x）**：原生的非阻塞 Java 操作，包括 `write_file`。
    *   **HTTP/网络（基于 Vert.x）**：通过 Vert.x `WebClient` 实现的稳健网络功能。
    *   **ToDo 与计划管理**：Agent 管理的内部任务列表。
*   **扩展工具（用户定义）**：
    *   **自定义实现**：框架应允许用户在 Java 中定义领域特定的工具。
    *   **发现**：扩展工具应通过类路径扫描（注解）或外部 JAR 加载进行注册。
*   **工具定义与 Schema**：
    *   **注解**：应使用 Java 方法上的 `@AgentTool` 注解定义工具。
    *   **自动 Schema**：系统应根据方法签名自动生成兼容 OpenAI 的 JSON Schema。
*   **沙箱与安全**：
    *   **范围限制**：基于 Bash 的工具应局限于项目根目录或指定的安全目录。
    *   **中断**：标记为 `@Sensitive` 的工具必须暂停以等待用户确认。

### 1.3 记忆与上下文（“大脑”）
*   **临时记忆**：系统应将所有交互（思考、工具调用、结果）记录到每日 Markdown 文件中 (`.ganglia/logs/YYYY-MM-DD.md`)。
*   **精选记忆**：系统应支持一个 `MEMORY.md` 文件，用于记录长期用户偏好和项目上下文。
*   **主动检索**：Agent 应能够搜索其自身的记忆文件（`grep` 日志或 `MEMORY.md`）以找回过去的信息。
*   **上下文管理**：系统应智能地修剪上下文窗口，以保持在 Token 限制内，同时保留最相关的信息。

### 1.4 人机交互
*   **规划阶段**：
    *   **提议**：对于复杂的请求，系统应在执行*前*生成结构化的执行计划（JSON 步骤列表）。
    *   **批准**：在继续执行前，系统应要求用户批准或修改计划。
*   **运行时中断**：
    *   **敏感操作**：执行 `@Sensitive` 工具（如 `delete`, `deploy`）必须触发阻塞式的用户确认提示。
    *   **用户交互**：Agent 应使用 `ask_selection` 展示选项列表或请求自由文本输入，并等待响应。
    *   **暂停/恢复**：系统应支持暂停 ReAct 循环以等待用户输入。

### 1.5 技能系统（“专业知识”）
*   **定义**：技能应是一个可打包的单元，包含：
    *   **领域知识**：专门的系统提示词、文档或启发式方法（例如“如何编写地道的 Go 代码”）。
    *   **专业工具**：针对该领域的 Java 工具（例如 `go_fmt`, `aws_deploy`）。
*   **生命周期管理**：系统应支持动态安装、激活和停用技能。
*   **上下文注入**：当技能处于活跃状态时，其相关的提示词和“最佳实践”应被注入到 Agent 的上下文中。
*   **发现**：Agent 应能够发现可用技能，并根据用户的任务推荐激活它们。

## 2. 非功能需求 (Quality Attributes)

### 2.1 可靠性与稳定性
*   **崩溃恢复**：系统应在每一步后将会话状态序列化到磁盘 (`.ganglia/state/`)，从而允许在不丢失数据的情况下从崩溃中恢复。
*   **速率限制**：系统应使用指数退避处理 API 速率限制（如 OpenAI 429 错误）。
*   **错误隔离**：工具执行失败不应导致主 Agent 循环崩溃。

### 2.2 易用性与体验
*   **延迟**：系统应支持向 UI 流式传输响应 (Token)，以最大限度地减少感知延迟。
*   **透明度**：Agent 的所有操作（“思考”、工具调用）必须实时对用户可见。
*   **可编辑性**：用户必须能够手动编辑生成的 `MEMORY.md` 和日志文件，以修正 Agent 的理解。

### 2.3 可观测性与调试
*   **追踪**：系统应为每个 ReAct 步骤和工具执行发出 OpenTelemetry 追踪。
*   **日志**：应提供原始提示词和 LLM 完成情况的详细日志（可配置），以便进行调试。
*   **审计追踪**：Markdown 日志文件应作为 Agent 所有操作的人类可读审计追踪。

### 2.4 安全性
*   **API 密钥管理**：API 密钥必须从环境变量或安全存储中加载，绝不记录或提交到 git。
*   **提示词注入**：系统应包含针对提示词注入攻击的基本防御措施（系统提示词指令）。
*   **沙箱隔离**：执行沙箱必须防止访问除允许的工作目录之外的主机系统。

### 2.5 可扩展性
*   **插件架构**：第三方开发者应能通过提供带有注解类的 JAR 来添加新工具。
*   **模型无关**：核心逻辑应通过 `ModelProvider` 接口与具体的 LLM 提供商解耦。
