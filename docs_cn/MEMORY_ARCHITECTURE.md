# Ganglia 记忆架构 (Memory Architecture)

> **理念**：“记忆即代码”。透明、用户可控、基于文件且分层。

## 1. 概述

Ganglia 实现了一个**三层记忆系统**，旨在平衡高保真推理（针对即时任务）与长期保留（针对项目上下文），同时高效地管理 Token 窗口限制。

## 2. 三层结构

### 第一层：短期记忆（轮次 - Turn）
*   **范围**：单次用户-Agent 交互循环（例如“修复错误 X”）。
*   **粒度**：极高。包含原始“思考”、准确的“工具调用”和完整的“观察结果”（如文件内容、命令输出）。
*   **存储**：`SessionContext` 中的内存 `Turn` 对象。
*   **生命周期**：仅在执行特定步骤时处于活跃状态。一旦步骤完成，即成为压缩候选对象。

### 第二层：中期记忆（上下文窗口 - Context Window）
*   **范围**：当前会话的历史记录。
*   **粒度**：混合。
    *   *近期轮次*：保留完整细节。
    *   *较早轮次*：根据 Token 限制进行修剪或总结。
*   **机制**：**带有语义修剪的滑动窗口**。
    *   `ReActAgentLoop` 自动修剪历史记录，以确保每次请求保留最多 **2000 tokens** 的对话上下文。
*   **与 ToDo 集成**：
    *   **计划/ToDo 列表**作为此层的支柱。
    *   当计划中的任务标记为 `DONE` 时，关联的轮次将被总结为简短的“结果”（例如“将 User.java 重构为 Record”）。
    *   原始轮次将从提示词上下文中移除，但会保存到磁盘。

### 第三层：长期记忆（项目知识 - Project Knowledge）
*   **范围**：跨会话的项目生命周期。
*   **存储**：
    *   `MEMORY.md`：精选的“经验教训”、架构决策和用户偏好。
    *   `.ganglia/logs/`：存档的过去会话的原始日志（可通过工具搜索）。
*   **检索**：**Agent 搜索**。Agent 在需要时使用工具（`grep_search`, `read_file`）主动从该层查找信息。

## 3. 压缩与总结策略

为了防止上下文溢出，Ganglia 采用了与任务生命周期挂钩的积极总结策略。

### “任务-轮次”周期
1.  **扩展**：用户给出目标 -> Agent 将其添加到 ToDo 列表。
2.  **执行**：Agent 执行多个轮次（推理 -> 行动 -> 观察）。这些轮次在上下文中累积。
3.  **完成**：Agent 将任务标记为 `DONE`。
4.  **压缩（钩子）**：
    *   **触发器**：任务状态变更为 `DONE`。
    *   **动作**：系统（或后台“反思”Agent）提取与该任务关联的所有轮次，并生成 1-2 句的总结。
    *   **替换**：原始轮次从 `SessionContext.previousTurns` 中移除，并由 ToDo 列表中的总结（如作为“结果”注释）或专门的“已完成任务”历史块替代。

## 4. 状态持久化

*   **会话状态**：每一轮次后序列化到 `.ganglia/state/session_ID.json`。包括 ToDo 列表、活跃变量和已压缩的历史记录。
*   **恢复**：加载会话将恢复 ToDo 列表和**已修剪**的上下文，而不是过去每个命令的完整原始历史（除非有明确要求）。

## 5. 记忆与工具集成

*   **`todo_complete`**：不仅仅是一个状态更新。它向记忆系统发出“打包此上下文”的信号。
*   **`remember`**：Agent 用于将重要事实写入 `MEMORY.md` 的特定工具（将其从短/中期记忆提升到长期记忆）。
