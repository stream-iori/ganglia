# Ganglia WebUI 设计原则 (UI-DESIGN-PRINCIPAL)

本文档定义了 Ganglia WebUI 的核心设计哲学与方法论，旨在指导开发者在面对复杂的“人机协同”场景时，能够做出一致、高效且符合用户直觉的 UI 决策。

---

## 🛠 四种分类方法对比矩阵

| 分类方法 | 核心切入点 | 关注的核心变量 | 适用阶段 (When) | 解决的核心问题 (Why) | 主要产出物 (What) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **一、 系统交互九宫格 (3x3 Matrix)** | 人机协同的系统状态机 | **系统风险状态** (绿/黄/红)<br>**用户认知负荷** (轻/中/重) | **架构设计期 / 核心 UI 框架搭建期** | 如何在不同风险下分配注意力？<br>什么时候该使用抽屉而非卡片？ | 核心页面布局 (主干/抽屉)<br>渲染策略 (虚拟滚动)<br>组件级状态机定义 |
| **二、 任务生命周期 (Lifecycle-based)** | 时间轴与操作漏斗 | **时间阶段** (事前/事中/事后)<br>**任务完成度** | **产品规划期 / 整体流程设计期** | 用户使用的第一步是什么？<br>做完任务后怎么验收？<br>Session 状态如何流转？ | 产品核心路径图 (User Flow)<br>Onboarding 流程<br>Review/验收视图设计 |
| **三、 目标与复杂度 (Intent-based)** | 业务意图与特定视图 | **业务目标** (查代码/写功能/修Bug)<br>**交互复杂度** | **功能迭代期 / 专属工具集设计** | 针对特定的高频场景，是否需要定制化的 UI 面板来提升效率？ | 专属视图组件 (如只读源码检视器、Diff 审核面板、高亮终端) |
| **四、 信息流向 (Information Flow)** | 契约与通信协议 | **数据流向** (人->Agent / Agent->人 / Agent->系统) | **前后端联调期 / API 设计期** | 前后端通过什么 Topic 通信？<br>JSON 结构怎么定？<br>异常怎么抛？ | EventBus Topic 列表<br>JSON Schema 定义<br>WebSocket/SockJS 断线重连策略 |

## 🔍 方法详述与场景应用

### 🎯 方法一：系统交互九宫格 (3x3 Matrix)
*—— 决定“骨架与肌肉”（UI 架构与渲染引擎）*

相比传统的四象限，九宫格通过引入“黄态（临界状态）”和“中等负荷（检视状态）”，精细化定义了人机协作的深度。

#### 🧩 交互组件九宫格映射表

| 用户认知负荷 (Y轴) \ 系统风险状态 (X轴) | 🟢 绿态：常规与安全<br>*(纯读、低耗时、预期内)* | 🟡 黄态：临界与警告<br>*(耗时长、初次写操作、资源高转)* | 🔴 红态：危险与异常<br>*(破坏性操作、API断连、报错)* |
| :--- | :--- | :--- | :--- |
| **极轻 (Glance)**<br>*(无需动手，余光感知)* | **【静默卡片】**<br>短命令 (如 `ls`)。<br>内联显示 1-3 行结果。 | **【动态指示器】**<br>耗时任务。<br>呼吸灯或打字机动效。 | **【局部错误容忍】**<br>Agent 尝试自愈。<br>卡片边框闪红。 |
| **中等 (Inspect)**<br>*(点击展开、滑动阅读)* | **【只读检视器】**<br>主动点文件名。<br>右侧抽屉 **Shiki 高亮**。 | **【深潜观测窗】**<br>海量日志喷涌。<br>**终端虚拟滚动面板**。 | **【错误溯源台】**<br>任务失败退出。<br>查阅 **Stack Trace**。 |
| **极重 (Block)**<br>*(强行阻断、必须决策)* | **【指令集输入】**<br>常规对话。<br>长段 Prompt 引导。 | **【人工授权表单】**<br>修改核心文件 (`ask_user`)。<br>**流暂停，强制决策**。 | **【上帝制动按钮】**<br>死循环或断网。<br>**全局 Kill Switch**。 |

#### 🚀 五大核心 Journey (基于九宫格映射)

1.  **巡航与脉搏 (Cruise & Pulse) [Glance + 🟢/🟡]**：系统静默运行，UI 保持安静。用户通过极简卡片和动态指示器感知“心跳”。
2.  **强力接管 (Active Intervention) [Block + 🟡]**：当 Agent 触碰敏感权限（写操作）时，通过模态化的授权表单强制用户介入，确保安全。
3.  **深度深潜 (Deep Dive) [Inspect + 🟢/🟡]**：用户对执行细节产生好奇。通过空间折叠（抽屉/侧边栏）查看长代码或海量 TTY 日志，不打断主流程。
4.  **异常自愈 (Self-healing) [Glance + 🔴]**：工程级轻微故障。UI 通过局部红框告警并提示 Agent 正在重试，无需用户操作。
5.  **紧急制动 (Emergency Stop) [Block + 🔴]**：系统失控或严重断连。全屏遮罩或高亮停止按钮，将控制权强制交还人类。

*   **最佳实践场景**：当你需要决定 **“这个终端输出是直接贴在聊天流里，还是必须单独做一个带虚拟滚动的抽屉”** 时，使用此九宫格定位其风险与负荷等级。

### ⏳ 方法二：任务生命周期 (Lifecycle-based)
*—— 决定“血管与神经”（整体流程与体验闭环）*

*   **分类维度**：从 Session 创建到结束的时间线。
*   **三大 Journey**：
    1.  **启动期 (Setup)**：选定工作区、选择 Persona、注入特定 Context（如阅读 MEMORY.md）、输入清晰指令。
    2.  **执行期 (Execution)**：Agent 分析、规划（Plan 模式）、调用工具执行（此阶段往往套用九宫格法则）。
    3.  **验收期 (Review & Teardown)**：任务结束，呈现全局 Diff 试图供人类 Review，引导人类将经验写入新的 Memory 并归档 Session。

---

### 🛠 方法三：目标与复杂度 (Intent-based)
*—— 决定“工具箱里的特殊工具”（定制化能力）*

*   **分类维度**：用户到底想干什么（读、写、修）。
*   **三大 Journey**：
    1.  **探索与答疑 (Read-Only)**：不改代码，重在搜索和高亮。需要极佳的代码阅读器 (Shiki) 和结构化 Markdown 渲染。
    2.  **构建与脚手架 (Create)**：从无到有。需要重点展示“新增文件树”和“结构对比”。
    3.  **重构与排错 (Modify & Debug)**：修改现有逻辑。极度依赖行级 Diff 视图和高亮报错日志。

---

### 🔌 方法四：信息流向 (Information Flow)
*—— 决定“接口与协议”（底层数据管道）*

*   **分类维度**：事件的发送者和接收者。
*   **三大 Journey**：
    1.  **Human ➔ Agent**：用户发送指令、中途打断补充上下文、回复 Agent 的求助表单（ask_selection）。
    2.  **Agent ➔ Human**：Agent 同步思维链 (Thought)、同步执行进度、或者抛出选项让用户做决定。
    3.  **Agent ➔ System**：调用外部工具（Bash、WebFetch），并将过程和结果（stdout/stderr）以旁路形式推送给前端 UI 用于监控。
