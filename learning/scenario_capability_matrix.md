# 场景与 Agent 能力矩阵 (MECE 校验版)

通过 MECE (相互独立、完全穷尽) 法则重新审视“软件工程生命周期”，我发现原矩阵中存在以下遗漏：
1. **输入阶段遗漏：** 缺乏项目从 0 到 1 的“认知冷启动”场景。
2. **生命周期遗漏：** 缺乏跨越单次会话的“休眠与唤醒”场景。
3. **输出维度遗漏：** 我们只考虑了代码交付，却忽略了当 Agent 遇到无法解决的问题时的“异常退出/交接”场景。

以下是补充完整后的全景交叉矩阵：

| 业务场景阶段 (Scenes) | 核心大脑动作 (Agent Core) | 强依赖的技能与工具 (Agent Tools) | 基础设施支撑 (Infra) | 关键衡量指标 (KPI) |
| :--- | :--- | :--- | :--- | :--- |
| **Init: 探底与知识库初始化**<br>*(认知冷启动，项目Onboarding)* | **全局扫盘:** 探测脚手架、配置文件和目录结构。<br>**规约提取:** 推断架构意图并生成项目级指南。 | **Tool(探测):** `list_directory`, `glob`。<br>**Bash:** 执行 `tree` 或查看依赖清单。 | **Store:** 建立初始的 `GEMINI.md` 或全局记忆。<br>**GW:** 擅长摘要的小模型。 | 探底生成的指南对后续任务代码风格的匹配度。 |
| **Pre: 计划确认与任务拆解**<br>*(需求理解、风险评估)* | **规划树构建:** 将复杂需求转化为有向无环图(DAG)。<br>**知识检索:** 搜索项目规约和历史记忆。 | **Sub(规划师化身):** 调用架构师 SubAgent。<br>**Tool(知识库检索):** 读取 `GEMINI.md` 和需求文档。 | **Store:** 向量检索与 RAG。<br>**GW:** 强推理模型 (如 o1 或 Claude-3.5-Sonnet) | 任务拆解的粒度是否足够小？是否识别了隐藏的依赖？ |
| **Mid(1): 代码搜索与理解**<br>*(寻找切入点)* | **语义导航:** 建立文件关联拓扑。<br>**动态注意力:** 裁剪无关代码，防止上下文溢出。 | **Tool(AST解析/Grep):** 全局 `grep_search`，`find_usages`。<br>**Bash:** 运行轻量级的结构分析脚本。 | **Ctx(上下文管理器):** 执行 `ContextCompressor` 滚动压缩。<br>**Box:** 只读沙箱环境。 | 目标文件命中率；Token 消耗量。 |
| **Mid(2): 代码编写与重构**<br>*(实施具体的修改)* | **精准外科手术:** 决定是全量重写还是局部替换。<br>**并发控制:** 多文件修改的顺序编排。 | **Tool(Patch/Write):** 使用 `apply_patch` 或 `replace_in_file` 实现幂等修改。 | **WS(Workspace):** Git 工作区隔离，打快照。<br>**GW:** 编码模型。 | 修改的原子性（不破坏未修改部分）；Diff 格式合法率。 |
| **Post(1): 验证与排错**<br>*(确保修改能跑通)* | **闭环排错:** 阅读长报错日志，提炼根因，重新决策。 | **Bash(执行环境):** 运行 `mvn test` 或 `npm run build`。<br>**Tool(日志分析器):** 提取 Error Stacktrace。 | **Box:** 带资源限制的执行沙箱（防死循环）。<br>**Dash:** 实时展示构建日志。 | 零人工干预下的编译通过率；Self-Correction 轮数。 |
| **Post(2): 部署与核任务**<br>*(代码合并)* | **意图打包:** 结构化提交，清理战场。 | **Bash:** 执行 `git commit -m` 和 `git push`。<br>**Sub(Git化身):** 生成 Conventional Commits。 | **WS:** 将隔离空间的修改合并到主分支。 | 提交信息的规范性；测试用例的覆盖。 |
| **Aux(1): 知识与记忆沉淀**<br>*(资产积累)* | **知识提炼:** 总结本次踩的坑和新引入的架构模式。 | **Tool(知识更新):** 追加写入项目的 `MEMORY.md`。 | **Store:** 触发长期记忆的持久化。 | 后续任务中避免重复犯错的概率。 |
| **Aux(2): 异常交接与退场**<br>*(无力解决时的Fallback)* | **现场保护:** 生成“我卡在了哪里”的分析报告，明确交接给人类。 | **Tool(人机通信):** `ask_user` 或生成异常报告。 | **Int(交互层):** 触发阻断式通知。 | 失败后人类接管的成本（上下文恢复时间）。 |
| **Ops: 断点续传与状态接管**<br>*(时间维度的管理)* | **状态序列化/反序列化:** 冻结或唤醒整套内存与沙箱树。 | **Tool(系统内置):** 底层的 Session Dump/Load 机制。 | **Ctx & Box:** 将进程状态落盘并能无损恢复。 | 会话重启后的状态一致性（不丢失上一步的进度）。 |

## MECE 视角下的完整性确认

通过这个扩充版本，我们用 MECE 思想审视：

1. **按时间轴 (Timeline) 穷尽了吗？**
   * 从初次接触项目 (`Init`) -> 接需求 (`Pre`) -> 找代码 (`Mid-1`) -> 写代码 (`Mid-2`) -> 验证 (`Post-1`) -> 提交 (`Post-2`) -> 总结 (`Aux-1`) -> 挂起/退出 (`Ops` / `Aux-2`)。**时间轴是闭环且穷尽的。**
2. **按成功/失败路径 (Happy/Unhappy Path) 穷尽了吗？**
   * 成功路径：`Pre -> Mid -> Post -> Aux`。
   * 失败重试路径：`Post(1)` 失败退回 `Mid(2)` (已在架构图的 Diag 中体现)。
   * 彻底失败/放弃路径：新增的 `Aux(2) 异常交接与退场`，确保 Agent 即使失败也能“优雅宕机”，而不是陷入死循环直到耗尽余额。
3. **按系统状态 (State Lifecycle) 穷尽了吗？**
   * 运行中状态：在 `Loop` 中。
   * 挂起/休眠状态：新增的 `Ops 断点续传`，应对了物理时间的流逝和网络的中断。

这个更新后的矩阵，不仅覆盖了理想状态下的“代码生成机器”，更描绘了一个具备“冷启动”、“记忆”、“优雅退场”和“抗物理中断”能力的**企业级数字工程师画像**。
