# Coding Agent 核心能力、辅助流程与场景推演指南

> **文档目的:** 深入探讨 Coding Agent 在“写代码”之外的辅助流程，并提供一套科学的方法论，用于确认 Agent 的核心能力边界与实战场景。

## 一、 Coding Agent 的五大核心辅助流程

一个成熟的 Coding Agent，纯粹的“代码生成”仅占工作量的 30%，其余 70% 由以下辅助流程构成，这些流程决定了交付的质量与安全性。

### 1. 认知与发现 (Cognitive Discovery)

- **符号导航 (Symbol Navigation):** 超越简单的文本搜索，理解类、方法、变量的引用关系（Find Usages）。
- **架构意图推断 (Architectural Intent):** 识别项目技术栈、编码风格（如响应式 vs 阻塞式）及核心逻辑边界。
- **风险探测:** 识别高危逻辑文件与边缘配置，评估修改的敏感度。

### 2. 策略规划与影响分析 (Strategic Planning)

- **任务拆解 (Decomposition):** 将模糊需求拆解为跨越数据库、后端逻辑、前端 UI 的具体步骤。
- **破坏性评估 (Breaking Change Analysis):** 预判修改是否破坏向下兼容性或引发级联错误。
- **方案对比 (Trade-off):** 内部模拟多种路径，选择对现有代码侵入性最小的方案。

### 3. 自动化验证与质量守卫 (Verification Loop)

- **闭环自愈 (Self-Healing):** 自动执行编译（mvn/npm），读取报错堆栈并进行多轮迭代修复直至通过。
- **测试生成与执行:** 自动编写单元测试并运行全量回归，确保不引入功能退化。
- **静态分析 (Linting):** 调用项目自带的格式化与校验工具（Eslint/Checkstyle），确保代码规范对齐。

### 4. 协作与知识外部化 (Externalization)

- **结构化提交 (Git Management):** 生成符合规范的 Commit Message 和详尽的 Pull Request 描述。
- **记忆持久化:** 自动更新 `MEMORY.md` 或 `ARCHITECTURE.md`，记录新引入的逻辑，防止后续遗忘。
- **交互式澄清:** 在需求模糊的关键节点主动停下询问用户，避免盲目假设。

### 5. 环境与生命周期治理 (Ops & Sandbox)

- **依赖治理:** 检查新依赖的安全漏洞与版本冲突（Dependency Hell）。
- **沙箱管控:** 自动化分支管理、数据库镜像配置，并在任务结束后清理临时资源。

## 二、 如何确认核心能力与关键场景？

架构师不应依靠直觉，而应通过以下四个维度系统性地锚定 Agent 的能力与场景。

### 1. 软件工程生命周期 (SDLC) 映射

- **逻辑:** 将资深工程师的一天进行“切片”，看哪些环节可被 Agent 认知化。
- **推演:** 需求理解 $\rightarrow$ 代码导航 $\rightarrow$ 架构设计 $\rightarrow$ 精准生产 $\rightarrow$ 质量守卫。

### 2. 工具与环境的“负荷测试”

- **逻辑:** 能力 = LLM 推理 + 外部工具链（LSP, Git, Shell, Debugger）。
- **推演:** 测试 Agent 对环境的支配力（如修复复杂的 `pom.xml` 冲突）和长链条任务的规划稳定性。

### 3. 维度矩阵：频率、风险与复杂度 (The 3D Matrix)

- **频率 (Frequency):** 该操作每天发生多少次？（高频操作如：写单元测试）。
- **风险 (Risk):** 做错后的爆炸半径？（高风险操作如：删除配置文件）。
- **复杂度 (Complexity):** 涉及的上下文深度？（高复杂度操作如：重构鉴权模块）。
- **筛选策略:** 优先攻克 **“高频 + 低风险 + 中等复杂度”** 的场景（Agent 的甜点位）。

### 4. 实证评测与黄金案例集

- **基准测试:** 利用 `SWE-bench` 等开源评测集测试真实 Issue 修复率。
- **黄金案例 (Gold-Sets):** 提取公司内部最常见的 20 个 Bug 修复过程，作为 Agent 能力确认的硬性指标。
- **失败根因分析 (RCA):** 分析失败是由于“感知失败”、“决策失败”还是“执行失败”，反向定义需补齐的能力。

## 三、 场景推演的进阶框架

除了能力边界，还可从以下视角深化场景设计：

1.  **认知负荷框架:** 测试高熵冲突（如文档与代码不一致）和低信噪比（海量日志找 Bug）下的表现。
2.  **自主度光谱框架:** 测试从“全自动黑盒”到“高频人机协作”不同模式下的状态对齐与指令优先级管理。
3.  **拓扑环境框架:** 测试在异构语言环境（Java + Node + Go）或受限沙箱环境（无外网/缺工具）下的适应力。
4.  **时间进化框架:** 测试长达数小时会话中的认知漂移（遗忘最初约束）及对陈旧知识的自主更新。

确认一个 Coding Agent
的核心能力（Capabilities）和关键场景（Scenarios），不是靠直觉，而是通过一套“自顶向下”的架构推演与“自底向上”的实证分析相结合的工程方法。

作为高级架构师，我会从以下四个维度来“锚定”这些能力和场景：

---

一、 自顶向下：软件工程生命周期 (SDLC) 映射
我们将传统的软件开发生命周期（SDLC）拆解，看哪些环节可以被 Agent “认知化”和“自动化”。

- 思考路径： 如果把一个资深工程师的一天切片，他都在做什么？
- 能力确认：
  - 需求理解力 (Requirement Analysis)： 将自然语言转化为技术规格（Schema, API 定义）。
  - 代码导航力 (Code Navigation)： 在数百万行代码中精准定位 Bug 或逻辑点（Grep, Find Usages）。
  - 架构设计力 (Architectural Reasoning)： 决定新代码应该放哪层，是否需要解耦。
  - 代码生产力 (Code Generation)： 精准的增量修改（Diff/Patch）。
  - 质量守卫力 (Quality Guard)： 编写测试、运行编译、修复 Lint。
- 场景确认： 每一个环节的极端情况就构成了关键场景（如：在一个没有文档的千万级代码库中定位 Bug）。

---

二、 自底向上：基于工具与环境的“负荷测试” (Capability vs. Tools)
能力不是虚无缥缈的，它取决于 Agent 能“握住”什么工具。

- 思考逻辑： 能力 = LLM 推理 + 外部工具 (LSP, Git, Shell, Debugger)。
- 能力确认：
  - 环境支配能力： Agent 能否正确配置 venv，能否修复 pom.xml 依赖冲突？
  - 长时任务规划能力： 当任务超过 50 个步骤时，Agent 是否会“断片”？
- 场景确认： 找那些“人类觉得繁琐但规则明确”的地方。
  - 场景示例： 跨 10 个微服务的接口重构（测试其长链条工具调用的稳定性）。

---

三、 维度矩阵：频率、严重性与复杂性 (The 3D Matrix)
这是确认优先级的核心框架。我们会把所有潜在场景放入一个三维坐标系：

1.  频率 (Frequency)： 这个操作每天发生多少次？（如：写 Unit Test 频率极高）。
2.  严重性/风险 (Criticality/Risk)： 如果 Agent 做错了，后果多严重？（如：删除数据库、写死循环逻辑风险极高）。
3.  复杂性 (Complexity)： 涉及到多少上下文？（如：修复单一 NPE 简单，重构整个鉴权模块复杂）。

- 核心场景筛选准则：
  - 高频 + 低风险 + 中复杂度： 这是 Agent 的“甜点位”（Sweet Spot），是必须具备的核心能力（如：日常 Bugfix、加注释、写样板代码）。
  - 低频 + 高风险 + 高复杂度： 这是架构师的“防御位”，是必须建立“意图防火墙”和“人工审批”的场景（如：核心架构重构）。

---

四、 实证与评测：SWE-bench 与 Internal Gold-Sets
最后，通过数据来“确认”和“校验”这些能力。

- 确认方法：
  1.  引入基准测试 (Benchmarking)： 使用类似 SWE-bench（基于真实 Github Issue 的评测集）来测试 Agent 在真实开源项目中的表现。
  2.  建立“黄金案例集” (Gold-Sets)： 在 Ganglia 内部，提取 10-20 个公司内部最常见的、最难搞的真实 Bug 修复过程。如果 Agent
      能自动化复现并修复这些 Bug，这项能力才算被“确认”。
  3.  失败根因分析 (Root Cause Analysis)： 当 Agent
      在场景中失败时，分析是“读不到代码（感知失败）”、“想错了（决策失败）”还是“写错了（执行失败）”，从而反向定义需要补齐的核心能力。

---

总结：Ganglia 的核心能力清单（示例）

基于上述框架，我们可以为 Ganglia 确认一份“核心能力对标场景表”：

┌─────────────────────┬──────────────────────────────────────────────┬──────────────────────────────────────────┐
│ 核心能力 │ 确认场景 (Scenarios) │ 确认标准 (Success Criteria) │
├─────────────────────┼──────────────────────────────────────────────┼──────────────────────────────────────────┤
│ 语义导航 (Nav) │ 在千万行代码中找到处理“订单超时”的具体函数。 │ 3 次以内查找命中目标文件。 │
│ 闭环修复 (Fix) │ 修复一个编译错误，并确保 mvn test 全部通过。 │ 自动修复编译错误且不引入新回归。 │
│ 增量重构 (Refactor) │ 将 5 个类中的冗余 Log 逻辑抽离成一个切面。 │ AST 结构正确，无语法错误，无冗余修改。 │
│ 安全执行 (Safety) │ 在隔离沙箱中运行不可控的 Python 脚本。 │ 脚本超时或越权时，沙箱能准确拦截并告警。 │
└─────────────────────┴──────────────────────────────────────────────┴──────────────────────────────────────────┘

架构师建议：
不要贪全。先确认 2-3 个高频且中等复杂度的能力（比如：基于 Grep 的 Bug 定位 + 基于 Patch 的局部修复），并把这几个场景做到 95%
以上的成功率，比在 100 个场景里只有 50% 的成功率要有用得多。
