# Human-in-the-Loop & Interaction Patterns

This document outlines strategies for involving users in the Agent's decision-making process, ensuring safety and alignment.

## Scheme 1: The "Plan First" Pattern (Recommended)

This pattern explicitly separates **Planning** from **Execution**. It answers the question: *"How do we get the TODO list?"* by delegating it to a specific LLM interaction **before** any tools are executed.

### 1. The Workflow

```text
User Request ("Build a website") 
       |
       v
[Planner Agent (LLM)]  <-- Specialized System Prompt
       |
       v
Generated Plan (List<Step>)  <-- JSON / Structured Output
       |
       v
[User Interface] --> User Reviews / Edits / Approves
       |
       v
[Executor Agent (ReAct)] <-- Executes approved steps one by one
```

### 2. How the TODO List is Obtained

The TODO list is **generated by the LLM** (acting as a "Planner"). We do not rely on hardcoded logic; we rely on the LLM's reasoning capabilities to decompose the task.

#### Step A: The Planner Prompt
We use a specific system prompt that disables tool execution and forces structured output.

> **System Prompt:**
> "You are an expert Project Planner.
> Your goal is to analyze the user's request and break it down into a logical sequence of execution steps.
> DO NOT execute any actions.
> DO NOT output conversational text.
> OUTPUT ONLY a JSON array of steps."

#### Step B: Structured Output (The Interface)
In Java, we define a POJO to capture this output.

```java
// Java Record for Structured Output
public record Plan(List<Step> steps) {}
public record Step(int id, String description, String reasoning) {}
```

#### Step C: The Planner Execution (Pseudo-code)
```python
def generate_plan(user_goal):
    system_prompt = "You are a planner... Return JSON matching { steps: [{id, description, reasoning}] }"
    
    # We force the LLM to return strict JSON
    response = llm.generate(
        system_prompt=system_prompt, 
        user_input=user_goal, 
        response_format={"type": "json_object"}
    )
    
    return parse_json(response) # Returns List<Step>
```

### 3. User Confirmation

Once the `List<Step>` is obtained:
1.  **Display:** Show the list to the user.
2.  **Edit:** Allow the user to remove steps, add steps, or edit descriptions (since it's just a Java List/JSON at this point).
3.  **Approve:** Once confirmed, this list becomes the **Context** for the Executor Agent.

## Scheme 2: Tool-Based Confirmation (Runtime)

*Alternative for smaller decisions.*

The Agent is allowed to run freely but must call a specific tool `ask_user(question)` before taking sensitive actions (like `delete_file`).

*   **Pro:** More dynamic.
*   **Con:** Harder to predict the full path of execution ahead of time.
