这是基于 Clawdbot 记忆系统的核心逻辑整理的结构化信息，旨在展示其如何实现本地化、持久化且高效的 AI 记忆。

### 1. 核心设计理念

- **本地化与所有权**：运行在本地计算机，而非云端；用户完全拥有数据。
- **文件即记忆**：记忆本体是纯文本 Markdown 文件，而非黑盒数据库。
- **混合搜索**：结合语义搜索（向量）和关键词搜索（BM25），确保查全率和查准率。
- **搜索优于注入**：不把所有记忆塞入 Context，而是按需搜索，节省 Token 和成本。

---

### 2. 记忆架构体系 (双层记忆)

Clawdbot 将记忆存储在 Agent 工作区（默认 `~/clawd/`）的 Markdown 文件中。

| 层级                  | 文件路径               | 用途                                              | 写入时机                     |
| --------------------- | ---------------------- | ------------------------------------------------- | ---------------------------- |
| **第 1 层：每日日志** | `memory/YYYY-MM-DD.md` | 记录当天的流水账、临时笔记、对话摘要。            | 全天持续追加写入。           |
| **第 2 层：长期记忆** | `MEMORY.md`            | 提炼后的事实、偏好、决策、经验教训。              | 当学习到重要信息时整理写入。 |
| **引导配置**          | `AGENTS.md`, `SOUL.md` | 定义 Agent 的指令、性格、以及如何使用记忆的规则。 | 每次会话启动时加载。         |

---

### 3. 读写机制 (工具链)

#### 读取记忆 (Read)

AI 不直接扫描所有文件，而是通过专用工具：

1. **`memory_search` (搜索)**：

- **功能**：在所有记忆文件中进行混合搜索。
- **参数**：`query` (查询词), `maxResults` (数量), `minScore` (相关度阈值，默认 0.35)。
- **原理**：并行运行 **向量搜索 (语义)** 和 **BM25 (关键词)**，通过加权公式合并结果：
  > `最终得分 = (0.7 * 向量得分) + (0.3 * 文本得分)`

2. **`memory_get` (获取)**：

- **功能**：在搜索定位后，读取特定文件的特定行范围。

#### 写入记忆 (Write)

- **无专用工具**：没有 `memory_write` 工具。
- **通用机制**：使用 Agent 标准的 `write` 和 `edit` 文件工具直接修改 Markdown 文件。
- **触发逻辑**：由 `AGENTS.md` 中的 Prompt 驱动（例如：“遇到重要决策时，写入 MEMORY.md”）。

---

### 4. 技术实现 (索引与存储)

后台自动处理文件变更，建立索引供搜索使用。

- **监听 (Watch)**：使用 Chokidar 监听文件变更（防抖 1.5秒）。
- **分块 (Chunking)**：将文件切分为约 **400 Token** 的块，块之间有 **80 Token 重叠**（保证上下文连贯）。
- **嵌入 (Embedding)**：调用 Embedding API（如 OpenAI/Gemini/Local）将文本块转为向量。
- **存储 (Storage)**：存入 SQLite 数据库（`~/.clawdbot/memory/<agentId>.sqlite`）。
- 利用 `sqlite-vec` 插件做向量搜索。
- 利用 `FTS5` 引擎做全文检索。

---

### 5. 上下文管理 (Context Optimization)

为了解决 LLM 上下文窗口限制（Context Window）和 Token 成本问题：

#### A. 压缩 (Compaction)

- **触发**：当上下文接近上限时。
- **动作**：

1. 保留最近 N 轮对话（完整）。
2. 将早期对话总结为摘要（Summary）。
3. 摘要存入 JSONL 历史记录，旧的详细对话从 Context 移除。

#### B. 记忆刷新 (Memory Flush) - _特色功能_

- **痛点**：压缩是有损的，可能丢掉细节。
- **机制**：在压缩发生**前**，系统强制 Agent 进行一轮“静默思考”。
- **指令**：“会话即将压缩，请审查对话并将重要事实写入 `memory/YYYY-MM-DD.md`。”
- **结果**：确保重要信息在被压缩成摘要前，已经固化到磁盘文件中。

#### C. 修剪 (Pruning)

- **对象**：工具产生的大量输出（如超长的构建日志、文件读取内容）。
- **策略**：
- **软修剪**：保留头尾（如前1500字符+后1500字符），中间截断。
- **Cache-TTL 修剪**：当 API 缓存（如 Anthropic Prompt Caching）过期时，主动修剪旧的历史记录以降低重新缓存的成本。

---

### 6. 多 Agent 与生命周期

- **隔离性**：
- 每个 Agent 拥有独立的工作区和 SQLite 索引文件。
- 默认情况下 Agent 只能搜索自己的记忆（软沙箱）。

- **会话生命周期**：
- 支持 `daily`（每日重置）、`idle`（闲置重置）模式。
- **会话挂钩 (Session Hooks)**：使用 `/new` 开启新会话时，系统自动提取上一个会话的最后 15 条消息，生成一个带有描述性文件名的 Markdown 存档，以便未来搜索。

### 总结

Clawdbot 的核心在于将 **LLM 的短期记忆（Context）** 转化为 **长期记忆（Markdown 文件）**，并通过 **SQLite** 进行高效索引。它通过“压缩前强制写入”机制解决了记忆丢失问题，通过“混合搜索”解决了检索准确性问题
