# Feature Specification: Refactor Observation Architecture

**Feature Branch**: `003-refactor-observation-arch`  
**Created**: 2026-03-11  
**Status**: Draft  
**Input**: User description: "重构观测架构，建立统一的 ExecutionContext 和 ObservationDispatcher，让所有的 Tool 和 Gateway 通过抽象接口汇报流式数据，然后让 WebUI 监听统一的事件流，解决 Token 和 TTY 流无法正确推送到前端的问题，同时提高系统的解耦程度。"

## Background & Motivation
Currently, Ganglia has two disjointed pipelines for agent observations:
1.  **Macro Pipeline**: Synchronous `AgentLoopObserver` invoked by the main `StandardAgentLoop` (captures Turn starts, Tool starts, etc.). `WebUIEventPublisher` currently attaches here.
2.  **Micro Pipeline**: Asynchronous EventBus publishing deep within Infrastructure classes like `BashTools` (for TTY streams) and `OpenAIModelGateway` (for Tokens).

Because `WebUIEventPublisher` only listens to the macro pipeline, it misses high-frequency streaming events (Tokens, TTY) published directly to the EventBus. To fix this without violating the Single Responsibility Principle by hardcoding EventBus logic into every tool, we need a unified observation architecture.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Unified Streaming to WebUI (Priority: P1)

As a WebUI user, I want to see the agent's thought process (tokens) and tool execution logs (TTY) stream in real-time so that I understand what the agent is doing without waiting for the turn to finish.

**Independent Test**: Start the WebUI with a streaming-enabled model. Issue a command like `run ls -la`. Verify that:
1. The "Thinking..." state shows tokens appearing sequentially.
2. The Tool Card's inspector shows terminal output lines arriving before the tool finishes.

**Acceptance Scenarios**:
1. **Given** the agent is generating a response, **When** tokens are received by the gateway, **Then** they are routed through the unified event stream and displayed in the WebUI.
2. **Given** a long-running tool is executing, **When** it produces stdout, **Then** the chunks are routed through the unified event stream and displayed in the WebUI immediately.

---

### User Story 2 - Decoupled Tool Implementation (Priority: P2)

As a developer creating new Tools or Skills, I want to report progress and errors through a clean context interface without needing to know about Vert.x EventBus internals.

**Independent Test**: Implement a dummy tool that emits intermediate progress. Verify it only depends on the core `ExecutionContext` interface and not Vert.x specific classes for event emission.

**Acceptance Scenarios**:
1. **Given** a new tool implementation, **When** it needs to emit a stream chunk, **Then** it can call `context.emitStream(chunk)` without referencing Vert.x.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Introduce an `ExecutionContext` (or similar interface) that provides standard methods for tools and gateways to emit events (e.g., `emitStream`, `emitError`).
- **FR-002**: Create an `ObservationDispatcher` responsible for taking events from the `ExecutionContext` and routing them to the underlying transport (Vert.x EventBus).
- **FR-003**: Refactor `AgentLoopObserver` and `WebUIEventPublisher`. The `WebUIEventPublisher` MUST act as a consumer of the unified EventBus topic rather than a direct synchronous observer of the loop.
- **FR-004**: Refactor `ModelGateway` implementations to use the new dispatcher/context for token emission instead of direct EventBus publishing.
- **FR-005**: Refactor `BashTools` and other relevant tools to use the new context for TTY stream emission.
- **FR-006**: Ensure `TraceManager` correctly captures all events from the new unified stream without duplication.

### Key Entities

- **ObservationDispatcher**: The central router for all observability events.
- **ExecutionContext**: The isolated context passed to executable components (Tools, Models) containing the session ID and emission methods.
- **ObservationEvent**: The standard data envelope for all events flowing through the system.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of Token and TTY events generated by the infrastructure reach the WebUI frontend via WebSocket.
- **SC-002**: Zero direct references to `Vertx` or `EventBus` exist within specific tool implementations (e.g., `BashTools` business logic) for the purpose of emitting observations.
- **SC-003**: System startup and existing end-to-end tests continue to pass with the new architecture.
- **SC-004**: Code complexity (coupling metrics) between Tools and the EventBus is demonstrably reduced.
