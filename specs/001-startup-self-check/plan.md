# Implementation Plan: Startup Self-Check & Config Initialization

**Branch**: `001-startup-self-check` | **Date**: 2026-03-10 | **Spec**: [specs/001-startup-self-check/spec.md](spec.md)
**Input**: Feature specification from `/specs/001-startup-self-check/spec.md`

## Summary

Implement a project startup self-check mechanism that validates the environment and ensures a default `.ganglia/config.json` is initialized if missing. This ensures a seamless "first-run" experience.

## Technical Context

**Language/Version**: Java 17  
**Primary Dependencies**: Vert.x 5  
**Storage**: File-based (.ganglia/config.json)  
**Testing**: JUnit 5  
**Target Platform**: Cross-platform (Java Runtime)  
**Project Type**: Agent Framework / CLI  
**Performance Goals**: Self-check completion < 100ms  
**Constraints**: Must be non-blocking (Vert.x event loop friendly)  
**Scale/Scope**: Local environment initialization  

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

- [x] **Hexagonal Layering**: Implementation will reside in the Infrastructure layer (File System) but triggered via API/Adapter layer during bootstrap.  
- [x] **Reactive Safety**: Will use Vert.x `FileSystem` (non-blocking) for all checks and creation.  
- [x] **Memory Transparency**: Configuration state is persistent; no direct impact on MEMORY.md but supports its foundation.  
- [x] **Control Flow**: Happens before the ReAct loop starts; maintains single loop focus.  
- [x] **Zero-SDK Compliance**: No external SDKs needed for file checks.  
- [x] **Tool-Groundedness**: Uses standard filesystem tools/APIs.
